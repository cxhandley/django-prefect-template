# Frontend Asset Pipeline

Tailwind CSS + DaisyUI are compiled locally via the Tailwind CLI. HTMX is vendored from npm.
`django-compressor` handles offline compression and cache-busting in staging/production.

## Directory layout

```
backend/
├── frontend/              # Node project — all build inputs live here
│   ├── package.json
│   ├── src/
│   │   └── main.css       # Tailwind v4 + DaisyUI v5 + custom styles
│   └── .gitignore         # ignores node_modules/
├── static/                # build outputs (gitignored, regenerated by npm run build)
│   ├── dist/
│   │   └── main.css       # compiled & minified CSS
│   └── vendor/
│       └── htmx.min.js    # copied from node_modules/htmx.org/dist/
└── staticfiles/           # Django collectstatic output (gitignored)
```

## Build scripts

```bash
# Install dependencies
cd backend/frontend
npm install

# One-shot build (minified)
npm run build

# Watch mode (dev — unminified, rebuilds on change)
npm run watch
```

`npm run build` does two things:
1. `tailwindcss -i src/main.css -o ../static/dist/main.css --minify`
2. `cp node_modules/htmx.org/dist/htmx.min.js ../static/vendor/htmx.min.js`

## Tailwind CSS configuration

`frontend/src/main.css` uses Tailwind v4 CSS-first configuration:

```css
@import "tailwindcss";
@plugin "daisyui";

/* Tell Tailwind where to find class names */
@source "../../apps";
```

No `tailwind.config.js` is needed — v4 configuration lives entirely in the CSS file.

## django-compressor

[django-compressor](https://django-compressor.readthedocs.io/) wraps `{% compress %}` template
tags to fingerprint and optionally minify static files at deploy time.

### Template usage

```html
{% load compress %}

{% compress css %}
<link rel="stylesheet" href="{% static 'dist/main.css' %}">
{% endcompress %}

{% compress js %}
<script src="{% static 'vendor/htmx.min.js' %}"></script>
{% endcompress %}
```

In **development** (`DEBUG=True`), `COMPRESS_ENABLED` defaults to `False` — the files are
served unchanged with no delay and no offline manifest needed.

In **staging/production** (`COMPRESS_OFFLINE=True`), run the offline compression step as part
of each deploy (see [Staging deploy](staging.md#deploy)):

```bash
python manage.py compress --force
```

This generates `staticfiles/CACHE/manifest.json` and pre-compressed/fingerprinted files that
WhiteNoise can serve with long `Cache-Control` headers.

## Docker: multi-stage build

`backend/Dockerfile` uses a two-stage build so no Node runtime ends up in the final image:

```
Stage 1 (node:20-alpine):  npm ci && npm run build
                           → /dist/main.css, /vendor/htmx.min.js

Stage 2 (python:3.13-slim): COPY --from=frontend /dist/ /app/static/dist/
                             COPY --from=frontend /vendor/ /app/static/vendor/
                             uv pip install ... && collectstatic
```

## docker-compose: frontend-build service

For local Docker-based development, `docker-compose.yml` includes a one-shot `frontend-build`
service that runs before the `web` service:

```bash
# Run the frontend build explicitly
docker compose run --rm frontend-build

# Or let it run automatically when starting the full stack
docker compose up -d
```

## Justfile commands

```bash
just frontend-build   # npm ci && npm run build (runs in devcontainer)
just frontend-watch   # npm run watch (live rebuild on template changes)
```

## CI

Add these steps before `collectstatic`:

```yaml
- name: Build frontend assets
  run: cd backend/frontend && npm ci && npm run build
```
